## 1. Problem Statement

A logged-in user has received a 6-character share code for a shared grocery list (generated by PRD-0003) and needs a way to join that list so it appears in their "My Lists" dashboard. Without a join flow, share codes are useless — users can only view lists they created. The join feature is the second half of the sharing primitive: create (PRD-0003) and join (this PRD) form the complete invite-by-code cycle.

## 2. Goals

- Allow a logged-in user to join an active shared list by entering its 6-character share code.
- Provide clear, specific error messages when the code is invalid, the list doesn't exist, the list is completed/deleted, or the user is already a participant.
- Add the user as a participant in `list_participants` on successful join, then redirect to the list view.
- Treat "already a participant" as a success case: redirect to the list view without creating a duplicate participant row.
- Notify existing SSE-connected viewers of the list when a new participant joins (event type `participant-joined`).
- Prevent brute-force enumeration of share codes via rate limiting (max 10 attempts per minute per user).
- Leverage the existing `list_participants` table schema and `UNIQUE(list_id, user_id)` constraint (per ADR-0003) for race-condition safety.

### Non-Goals

- Creating lists (PRD-0003).
- Invitation emails or external notification when someone joins a list.
- Requesting to join a list (approval flow) — flat access model per ADR-0003 means anyone with the code joins immediately.
- Permission levels or roles after joining — flat access per ADR-0003.
- Leave/unjoin a list (deferred to a future PRD).
- Listing participants on the list view (deferred to a future presence feature).
- Search for lists by name or other criteria.
- Generating or sharing codes (PRD-0003).
- QR code generation for share codes.

## 3. User Stories

- As a user with a share code, I want to enter the code on the dashboard so that the shared list appears among my lists.
- As a user who enters an invalid (non-6-char) code, I want to see an error message so that I know the format is wrong.
- As a user who enters a valid code for a non-existent list, I want to see "List not found" so that I can verify the code with the person who shared it.
- As a user who enters a valid code for a completed list, I want to see "This list has been completed" so that I understand why I can't join.
- As a user who enters a code for a list I already belong to, I want to go directly to the list view without an error.
- As a participant on a list, I want to see when someone new joins so that I know who else is viewing the list.
- As a user, I don't want to be able to brute-force-guess share codes so that my lists remain private.

## 4. Detailed Requirements

### 4.1 Functional Requirements

| ID | Requirement | Priority | Notes |
|----|-------------|----------|-------|
| FR-1 | The system shall provide a "Join List" form on the dashboard (`/`) with a single text input for the share code and a "Join" submit button. | must | |
| FR-2 | The system shall validate that the submitted code is exactly 6 alphanumeric characters (a-z, 0-9, case-insensitive). Codes shorter than 6, longer than 6, or containing non-alphanumeric characters shall be rejected with an error message "Code must be 6 alphanumeric characters." | must | Case-insensitive: normalize to lowercase before validation and lookup. Leading and trailing whitespace shall be trimmed before validation and lookup. Processing order: trim → lowercase → validate → lookup. |
| FR-3 | The system shall look up the list by `code` (case-insensitive, normalized to lowercase) in the `lists` table. | must | |
| FR-4 | If no list with the given code exists (including lists with `status='deleted'`), the system shall return an error message "No list found with this code." | must | Deleted lists are not joinable; treat them as non-existent. |
| FR-5 | If the list exists but has `status='completed'`, the system shall return an error message "This list has been completed and is no longer accepting new members." | must | |
| FR-6 | If the list exists and is active, the system shall check whether the user is already a participant via `SELECT 1 FROM list_participants WHERE list_id = ? AND user_id = ?`. | must | |
| FR-7 | If the user is already a participant on an active list, the system shall redirect to `/list/:code` without creating a duplicate participant row. | must | Not an error state. The `UNIQUE(list_id, user_id)` constraint on `list_participants` is the final safeguard. |
| FR-8 | If the user is not yet a participant on an active list, the system shall INSERT a row into `list_participants` (list_id, user_id), redirect to `/list/:code`, and broadcast a `participant-joined` SSE event to existing viewers of that list. | must | Use try/catch for the UNIQUE constraint violation to handle race conditions gracefully. |
| FR-9 | The system shall rate-limit join attempts per user: maximum 10 POSTs to the join endpoint per rolling 60-second window. On exceeding the limit, return HTTP 429 with error message "Too many attempts. Please try again in one minute." | must | This prevents brute-force enumeration of share codes. |
| FR-10 | The system shall broadcast an SSE event with type `participant-joined` to all SSE-connected viewers of the list when a new participant joins. The event payload shall include `{list_id: N, user_id: N}`. | must | Per ADR-0006: SSE via http-kit async handlers. Future presence features may extend this payload with `display_name`. |
| FR-11 | The "Join List" form shall be accessible from the "My Lists" dashboard (the home page for authenticated users), positioned near the "Create List" button for discoverability. | must | |
| FR-12 | Both "Join List" and the existing "Create List" (FR-1 of PRD-0003) shall be co-located at the top of the dashboard page. | must | |
| FR-13 | The join-code form shall support keyboard submission (Enter key). | must | |
| FR-14 | Error messages shall include `role="alert"` for screen reader accessibility. | must | |

### 4.2 Data Requirements

Per ADR-0003 and PRD-0003, the join feature uses these existing tables:

- **`lists`:**
  - `id` SERIAL PRIMARY KEY
  - `code` VARCHAR(6) UNIQUE NOT NULL — the share code the user enters (lowercase a-z0-9)
  - `name` VARCHAR(100) NOT NULL
  - `status` VARCHAR(20) NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'completed', 'deleted'))
  - `version` INTEGER NOT NULL DEFAULT 1 — optimistic locking (not modified by join)
  - `created_by` INTEGER REFERENCES users(id) ON DELETE SET NULL
  - `created_at` TIMESTAMPTZ DEFAULT now()
  - `completed_at` TIMESTAMPTZ — NULL until completed
  - `updated_at` TIMESTAMPTZ DEFAULT now()
  - INDEX on (`code`) — used for join lookup

- **`list_participants` (the table we INSERT into):**
  - `id` SERIAL PRIMARY KEY
  - `list_id` INTEGER NOT NULL REFERENCES lists(id) ON DELETE CASCADE
  - `user_id` INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE
  - `joined_at` TIMESTAMPTZ DEFAULT now()
  - UNIQUE(`list_id`, `user_id`) — race condition safeguard

**No new tables or columns are required for this feature.** The `list_participants` table already exists per ADR-0003 and PRD-0003.

### 4.3 UI / UX Requirements

- **Dashboard (`/` for authenticated users) — "Join List" form:**
  - Positioned at the top of the dashboard, alongside the "Create List" button (from PRD-0003).
  - Single text input: placeholder "Enter share code", auto-capitalize off, maxlength 6, input mode "text".
  - "Join" button next to the input.
  - On successful join: redirect to `/list/:code` — the full list view.
  - On error: display an inline error message below the form (not a redirect). Error text is one of:
    - "Code must be 6 alphanumeric characters."
    - "No list found with this code."
    - "This list has been completed and is no longer accepting new members."
    - "Too many attempts. Please try again in one minute."
  - No loading spinner required (HTMX swap is near-instant).
  - After successful join, when the user navigates back to `/`, the new list appears in the dashboard (standard page re-render).

- **Already a participant flow:**
  - Not an error state. The user submits the form, and the system redirects to `/list/:code` silently.
  - No notification displayed to the user (they see the list they wanted).

- **SSE notification (invisible to new joiner, visible to existing viewers):**
  - No dedicated UI element shows "User x joined". The participant-joined event is consumed by the server-side SSE handler but currently results in no user-visible UI change (presence display is deferred).
  - The SSE infrastructure must emit the event so that future presence features can consume it.

- **Rate limit behavior:**
  - The form remains visible but returns a 429 error page/component with the rate-limit message.
  - After 60 seconds, the user can attempt again (standard rate-limit window reset).

- **Keyboard and accessibility:**
  - Form shall be submittable via Enter key on the code input.
  - Error messages shall have `role="alert"` for screen readers.
  - Input shall have an associated `<label>` (not a placeholder-only label).

### 4.4 Non-Functional Requirements

| ID | Requirement | Target |
|----|-------------|--------|
| NFR-1 | Join flow (code validation, participant check + insert, redirect) must complete in < 500 ms (p95). | performance |
| NFR-2 | Rate limiting must be enforced per-user, not per-IP, using in-memory state (e.g., a Clojure `atom` with a map of `user-id -> [timestamps]`). | security |
| NFR-3 | Rate limit window precision: sliding window of 60 seconds with at least second-level granularity. | security |
| NFR-4 | The `UNIQUE(list_id, user_id)` constraint is the final defense against race conditions — no application-level locking is required. | data integrity |
| NFR-5 | SSE `participant-joined` events must reach all connected viewers within 2 seconds of the insert. | real-time |
| NFR-6 | Code validation and lookup must be case-insensitive: "ABC123" and "abc123" map to the same code. Codes are always stored lowercase. | usability |

## 5. Out of Scope

- Leaving/unjoining a list (deferred).
- Invitation emails when someone shares a code.
- Request/approval join flow (flat access per ADR-0003).
- Participant presence UI on the list view (deferred).
- Listing participants or showing participant count on the list view.
- QR code generation for share codes.
- "Share this list" action from the UI (the code is already copyable per PRD-0003).
- Rate limiting on other endpoints (only join endpoint).
- IP-based rate limiting (user-based only).
- Persistent rate-limit state (in-memory only, reset on server restart).

## 6. Acceptance Criteria

- [ ] **AC-1 (Join active list):** Given a logged-in user on the dashboard with a valid 6-char share code for an active list they are not yet a participant of, when they submit the join form, then they are added to `list_participants` and redirected to `/list/:code`.
- [ ] **AC-2 (Already a participant):** Given a logged-in user who is already a participant of an active list, when they submit the join form with that list's code, then they are redirected to `/list/:code` with no duplicate participant row created.
- [ ] **AC-3 (Race condition — duplicate join):** Given two simultaneous join requests from the same user for the same list, when both requests are processed concurrently, then exactly one participant row is created (enforced by `UNIQUE(list_id, user_id)`), and the user is redirected to `/list/:code`.
- [ ] **AC-4 (Invalid code format):** Given a logged-in user, when they submit a code that is not exactly 6 alphanumeric characters, then they see the error "Code must be 6 alphanumeric characters."
- [ ] **AC-5 (Code not found):** Given a logged-in user, when they submit a valid-format code that does not correspond to any list (or corresponds to a deleted list), then they see the error "No list found with this code."
- [ ] **AC-6 (List is completed):** Given a logged-in user, when they submit a valid code for a completed list, then they see the error "This list has been completed and is no longer accepting new members."
- [ ] **AC-7 (Rate limiting):** Given a logged-in user who has made 10 join attempts in the past 60 seconds, when they make an 11th attempt, then they receive HTTP 429 with the error "Too many attempts. Please try again in one minute."
- [ ] **AC-8 (Rate limit window):** Given a user who has been rate-limited, when 60 seconds have passed since their 10th attempt, then they can successfully submit a join request again.
- [ ] **AC-9 (SSE participant-joined):** Given two logged-in users (User A and User B) viewing the same active list via SSE, when User B joins normally (is not yet a participant), then User A receives an SSE event with type `participant-joined` containing `{list_id, user_id}` within 2 seconds.
- [ ] **AC-10 (Case insensitivity):** Given a list with code `abc123`, when a user submits code `ABC123`, then the join succeeds (lookup is case-insensitive).
- [ ] **AC-11 (Case normalization):** Given a user submits code `AbC123 ` (mixed case), when the system processes it, then the code is normalized to lowercase `abc123` before lookup.
- [ ] **AC-12 (Dashboard form visible):** Given a logged-in user on the dashboard `/`, then the "Join List" form is visible with a text input and "Join" button.
- [ ] **AC-13 (Form submission via Enter, FR-13):** Given the join form, when the user presses Enter in the code input, then the form submits.
- [ ] **AC-14 (Error accessibility, FR-14):** Given an error message is displayed on the join form, then it has `role="alert"` for screen readers.
- [ ] **AC-15 (Dashboard after join, depends on PRD-0003 dashboard functionality):** Given a user who has successfully joined a list, when they navigate back to the dashboard `/`, then the newly joined list appears on their dashboard.
- [ ] **AC-16 (Join and Create visibility):** Given a logged-in user on the dashboard page `/`, when the page loads, then both the "Join List" form and the "Create List" button are visible at the top of the page.

## 7. Open Questions

- **Q1:** Should SSE `participant-joined` events be consumed immediately for any UI indication, or just emitted for future use? — **Answer: Emit only; no UI yet.** The event infrastructure is set up now so that a future presence feature can consume events without backend changes.
- **Q2:** Should rate limiting be in-memory (lost on restart) or persisted to PostgreSQL? — **Answer: In-memory.** This is a safety measure, not a critical security boundary. PostgreSQL persistence is overkill; a server restart resets counters, which is acceptable.
- **Q3:** Should deleted list codes be indistinguishable from non-existent codes (same error), or should we give a specific error? — **Answer: Same error "No list found with this code."** This avoids leaking information about which codes previously existed.
- **Q4:** Should we normalize codes to lowercase before storing and before lookup in the `lists` table's JOIN flow (PRD-0003 generates codes lowercase + digits already)? — **Answer: Yes.** The join input normalizes to lowercase before validation and DB lookup. `lists.code` is always lowercase in the database per PRD-0003 FR-9.
- **Q5:** Should the rate limit counter reset on successful join, or count attempts regardless of outcome? — **Answer: Counts all POSTs to the join endpoint regardless of outcome.** Otherwise a user could brute-force by interspersing guesses with dummy successful joins.
- **Q6:** What happens to `list_participants` when a list is deleted or completed? — **Answer: Rows remain.** `list_participants` has `ON DELETE CASCADE` from `lists`, so if the list row is actually deleted (not soft-deleted), participants cascade. For soft-delete (`status='deleted'`), participant rows are retained.
- **Q7:** Should we show a success message or animation when the user joins a list before redirecting? — **Answer: No.** Redirect to the list view immediately. The list appearing on the dashboard after redirect is the success signal.
